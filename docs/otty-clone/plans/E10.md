## TL;DR ‚Äî shape of the work

Almost everything is **client-side over the libghostty cell grid + existing engines**; the **one
wire/RPC decision point** is host-side **Open / Reveal-in-Finder** (the file lives on the host Mac,
not the client). Per the carryover, that is delivered by **extending the existing E4 metadata RPC**
(`metadataRequest` type 16 / `metadataResponse` type 30) with **two new `MetadataVerb` bytes
(`.openPath = 9`, `.revealPath = 10`)** ‚Äî NO new wire message type, NO new socket ‚Äî plus a thin
macOS-only `HostPathActionPerformer`. `Copy Path` is a client pasteboard write; `Change Directory
Here` injects `cd <path>\n` as **verbatim UTF-8** down the existing terminal input path.

The status bar, ‚åò-hold underline, Jump-To, Hint Mode, and Command Navigator are pure-model +
SwiftUI overlays mounted in `TerminalLeafView` / `OverlayCoordinator`. cwd comes from
`PaneSpec.lastKnownCwd` (already updated from OSC 7 at `WorkspaceStore.swift:4388`), last-exit from
`TerminalViewModel.lastCommand` (set from `commandStatus(.idle(exitCode:durationMS:))`), pane-kind
from `PaneSpec.kind`, host from the app-global connection ‚Äî all already client-side, **no wire change
for the status bar**.

## The geometry seam (needed by WI-5 underline + WI-9 hint labels)

The pure detector works on `[String]` lines, but rendering an underline / a 2-letter label at the
exact cell needs a **viewport row‚Üípixel mapping**. `scrollbackTextLines()` is a flat mirror with no
viewport/coordinate info. **WI-2 adds a small optional capability** so production renders faithfully
and headless stays inert (honest ceiling, no faked overlay):

```swift
// TerminalSurface.swift ‚Äî new optional capability (separate protocol, like TerminalSurfaceActions)
public struct TerminalCellMetrics: Sendable {        // points, not pixels
    public var cellWidth: CGFloat, cellHeight: CGFloat
    public var cols: Int, rows: Int                  // visible viewport grid
    public var originX: CGFloat, originY: CGFloat     // viewport top-left in the view's coord space
}
public protocol TerminalViewportSnapshotting: AnyObject {
    /// Visible viewport rows top‚Üíbottom (NOT scrollback) for ‚åò-hold underline + hint labels.
    func viewportTextRows() -> [String]
    /// nil ‚áí no live surface (headless/placeholder) ‚áí overlays simply do not render.
    func cellMetrics() -> TerminalCellMetrics?
}
```

`GhosttySurface` conforms (WI-2) using its existing `cellWidthPx/cellHeightPx/contentScale` (already
captured) + a viewport text read (mirror of `scrollbackTextLines()` restricted to the visible grid)
+ the `ghostty_action_cell_size_s` it already receives. Headless/`BuildStatusPlaceholderView`
returns `nil` ‚áí underline/labels are absent (never wrong). A `(row,colStart..<colEnd)` match maps to
`CGRect(x: originX + cellWidth*colStart, y: originY + cellHeight*row, ‚Ä¶)` ‚Äî plain `*`+`+`.

### Accepted ceiling ‚Äî OSC 8 hyperlink runs are NOT surfaced in Hint Mode / Jump-To (audit `osc8-hyperlinks-not-in-hint-or-jumpto`)

`hint-mode.md` + `files-and-links.md` list "OSC 8 hyperlinks" among the hintable / jump-to target
forms. The clone surfaces ONLY the plain-text `TerminalLinkDetector` regex scan (paths/URLs whose
**display text** is itself a URL/path) into `HintLabelAssigner.targets` / `JumpToModel.items`; an OSC 8
link whose display text is NOT a URL (e.g. the text `click here` linking to `https://‚Ä¶`) gets **no**
hint label and **no** Jump-To row.

This is a **renderer-target-only (libghostty) gap, not reachable from the headless seam**, confirmed at
the C ABI: the viewport snapshot seam reads text via `ghostty_surface_read_text` ‚Üí `ghostty_text_s`,
whose fields are `tl_px_x/tl_px_y/offset_start/offset_len/text/text_len` ‚Äî **no hyperlink/URL field**.
The ONLY OSC 8 URL access in the entire vendored `ghostty.h` is `GHOSTTY_ACTION_MOUSE_OVER_LINK`
(`ghostty_action_mouse_over_link_s { url; len }`), a **hover** callback for the single link under the
**mouse** ‚Äî it does NOT enumerate the OSC 8 link runs across the viewport cell grid. So merging OSC 8
runs into the keyboard hint/jump target set would require a NEW libghostty C API (per-cell hyperlink
ID/URL enumeration) + a vendored-fork patch + an xcframework rebuild (the fragile Zig/SDK step), none
of which is reachable from `AislopdeskWorkspaceCore` where `HintLabelAssigner`/`JumpToModel` live.
**Accepted as an honest gap** (recorded in `docs/DECISIONS.md`): the libghostty hover-underline +
‚åò-click open path still covers OSC 8 links for the mouse; only the keyboard hint/jump surfaces miss the
display-text-isn't-a-URL case. Revisit if/when a host- or renderer-side cell-hyperlink read API lands.
