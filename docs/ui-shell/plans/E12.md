# E12 — Composer + Prompt Queue (implementation plan)

> Goal: mount the built `InputBar`/`InputBoxModel` composer at the bottom of the focused pane, add
> draft persistence + pin + a non-activating float panel (macOS) / bottom-sheet (iOS), and the
> Prompt Queue (`⌘⇧M`) with chips + OSC-133 idle dispatch. **All client-side. No wire change.**

## Binding constraints carried into this plan

From `docs/ui-shell/plans/E12-carryovers.md` (HARD exclusions — each is an acceptance criterion):

- **Agent-AGNOSTIC plumbing only.** The composer + queue target the **active pane's session**, full
  stop. Do NOT add any agent picker / per-agent backend selection / codex/opencode affordance, even
  though the spec's screenshots show "OpenCode". The float-panel title reads **"SlopDesk Composer — Claude
  Code"** only when the active pane's `claudeStatus != .none`; otherwise **"SlopDesk Composer"** (no
  agent-name guessing). No multi-agent chooser anywhere.
- **No horizontal/top tab-bar** introduced anywhere E12 touches (vertical-tabs-only, `docs/DECISIONS.md`).
- **No SSH-host filter/pill** in any surface E12 touches.
- **No agent supervision / status / approval / send-to-chat UI.** Those are **E13** (`.sendToChat`
  stays a no-op stub here). E12 ships ONLY the composer + prompt-queue input mechanics. Reading/queuing
  is not agent-driving; keep the OSC-133 idle-dispatch seam, surface no supervision chrome.

## Wire / iOS / float-codec flags

- **Touches the WIRE protocol / golden corpus?** **NO.** Composer + queue are a client-side UI overlay.
  A sent/dispatched prompt rides the EXISTING per-pane ordered-OUT path (`InputBarModel.sendSink` →
  `TerminalViewModel.sendInput` → wire type 3), byte-identical to a typed line. No new wire type, no
  message-set change, no `golden/golden_vectors.json` edit, no `docs/20-wire-protocol.md` edit.
- **Touches iOS sources (`#if os(iOS)`)?** **YES.** The bottom-sheet composer + iOS rich-paste reader are
  `#if os(iOS)`; the NSPanel float + AppKit rich-paste reader are `#if os(macOS)`. iOS rots silently on a
  macOS `swift build` — **run `bash scripts/check-ios.sh` after WI-5/WI-6.**
- **Touches float-math / codec / controller code?** **NO.** No FEC/packetizer/controller/float math; no
  `addingProduct`/`fma`; no NaN min/max sites. Pure UI + view-model logic.

**Architecture mapping decision (host+client split, stated per the prompt):** A single "next idle
prompt" trigger has NO single equivalent here because Claude Code runs in the **alt-screen** (it emits no
OSC-133 prompt marks), while a normal shell does. So the dispatch trigger is the union of TWO faithful
signals, resolved per-pane:

1. **Normal terminal pane** → client `modeTracker` emits `.promptStart` (OSC-133;A) when the shell is
   back at an idle prompt. This is the literal idle-prompt trigger for that source.
2. **Agent (alt-screen) pane** → `claudeStatus` transitions to `.idle` (host-detected, wire type 27,
   already wired through `LivePaneSession.feedAgentSignal`). This is the faithful equivalent of "the
   agent went idle between turns".

`PromptQueueModel.dispatchNext()` is the single sink both signals call. Documented in `ComposerModel`.

## Approach per acceptance story (+ verification)

### ES-E12-1 — `⌘⇧E` opens a multi-line composer; `↩`/`⇧↩` newline, only `⌘↩` sends. — both
- Route `.composer` → `requestComposerInActivePane()` → `ComposerModel.toggle()` shows `ComposerBar`
  at the focused pane's bottom (mount precedent: find bar). `ComposerBar` reuses `InputBar`'s
  `axis: .vertical` editor; bare `↩`/`⇧↩` insert newlines (already the `richMode` behavior in
  `InputBar.onKeyPress(.return)`), `⌘↩` calls `composer.sendDraft()`.
- **Verify:** unit (`ComposerModelTests`: `toggle` flips `isVisible`; `sendDraft` writes `draft + CR`
  once to the send-spy then clears + hides; a bare-newline draft is preserved un-sent) + GUI
  (`scripts/check-macos.sh`: ⌘⇧E shows the bar, multi-line typing, ⌘↩ injects).

### ES-E12-2 — `⎋` cancels keeping draft; draft survives tab switches. — both
- `cancel()` hides but does NOT clear `draft`. `ComposerModel` lives on the durable `LivePaneSession`
  (panes mount at opacity-0, never torn down per memory), so the draft persists across tab switches and
  is restored on `open()`.
- **Verify:** unit (`cancel` leaves `draft` intact, `isVisible == false`; re-`open` shows same draft) +
  GUI (type, ⎋, switch tab, ⌘⇧E → draft restored).

### ES-E12-3 — `⌘V` rich (HTML/RTF/image → Markdown); `⇧⌘V` plain. — both
- `ComposerBar` intercepts `⌘V`/`⇧⌘V`. `⌘V` reads NSPasteboard/UIPasteboard HTML/RTF (or an image →
  Markdown image ref) and runs `RichPasteMarkdown.markdown(fromHTML:)`, inserting Markdown at the
  cursor; `⇧⌘V` inserts the plain-string item verbatim. Reuses the existing `onPasteToComposer` seam
  for the context-menu "Paste and continue in Composer" path.
- **Verify:** unit (`RichPasteMarkdownTests`: `<h1>`/`<strong>`/`<a>`/`<ul><li>` → `#`/`**`/`[..](..)`/`-`)
  — the platform read is GUI-only, so the pure converter carries the test weight. GUI smoke for the
  actual paste.

### ES-E12-4 — pin keeps composer across tab switches; float = non-activating panel (macOS), bottom-sheet (iOS). — GUI-verifiable
- **Pin:** `isPinned` promotes the `ComposerBar` to a WINDOW-level mount (above the tab/pane switcher),
  persisted via `composerPinned` pref. **Float:** `isFloating` opens `ComposerFloatPanel`
  (`.nonactivatingPanel` + `.floating` level + `becomesKeyOnlyIfNeeded`), keeping the SAME `ComposerModel`
  so `⌘↩` injects into the originating pane; send/close docks back. **iOS:** pin re-presents the
  `ComposerSheet` across navigation; float is a documented no-op (`ComposerSheet` only). Title:
  "SlopDesk Composer — Claude Code" only when `claudeStatus != .none`, else "SlopDesk Composer".
- **Verify:** GUI (`scripts/check-macos.sh`: pin → switch tab → composer rides along; float → window
  stays on top WITHOUT activating the app / claiming the menu bar; type elsewhere then ⌘↩ injects to the
  origin pane). iOS: `scripts/check-ios.sh` compiles the sheet; Maestro/manual sheet smoke.

### ES-E12-5 — `⌘⇧M` opens prompt queue; `⌥⌘↩` enqueues; chips reorder/edit/remove; each fires at next idle in order. — both
- Route `.promptQueue` → `requestPromptQueueInActivePane()` opens the composer in queue-input mode
  (placeholder "Add to Prompt Queue…", `↩` adds a line). `⌥⌘↩` in the composer → `enqueueDraft()` splits
  the draft into one `PromptQueueModel` item per non-blank line. `PromptQueueStrip` chips:
  tap = `editChip` (pop back into composer), `✕` = `removeChip`, drag = `moveChip`. Dispatch: BOTH idle
  signals (normal-pane `onPromptIdle` OSC-133;A, agent-pane `claudeStatus → .idle`) call
  `notePromptIdle()` → `PromptQueueModel.dispatchNext()` writes the next item's bytes through the pane
  send sink, in order, one per idle.
- **Verify:** unit (`PromptQueueModelTests`: multi-line enqueue → N items in order; `move`/`remove`/`take`;
  `dispatchNext` pops FIFO and returns `nil` when empty; strip hidden when empty. `ComposerModelTests`:
  `enqueueDraft` appends to queue + does NOT call send; `notePromptIdle` writes exactly the head item's
  bytes once per call, in order) + GUI smoke (queue two lines, watch them fire as the shell returns to
  prompt).

## Notes / traps honored
- Manual ordered-OUT only: composer sends through `InputBarModel.sendText`/`sendRaw` → the ONE per-pane
  drain (never a new `Task { await client.sendInput }`, never a second socket) — the unstructured-Task
  reorder class.
- Validate-then-degrade: `RichPasteMarkdown` and queue parsing never crash on hostile/empty clipboard or
  draft input; no force-unwrap.
- No app-layer crypto/auth; queue + drafts are client-local raw text, never transmitted as a new wire type.
- iOS rots silently → `check-ios.sh` is a REQUIRED gate after WI-5/WI-6.
- NSPanel: `.nonactivatingPanel` + `becomesKeyOnlyIfNeeded` + `level = .floating` so the float stays on
  top WITHOUT activating the app or claiming the menu bar; panel keeps the origin
  pane's `ComposerModel` so `⌘↩` injects to the correct PTY.
