## Binding constraints folded in (from E15-carryovers.md + CLAUDE.md)

- **VERTICAL-TABS-ONLY (hard).** The theme editor's chrome regions must NOT introduce a "Tabbar" region
  that implies a horizontal/top tab strip, and must NOT add any theme key/swatch whose only purpose is a
  top tab bar. The reference screenshot `dark-mode-theme.png` shows a **Tabbar** swatch group — we MAP it to the
  vertical rail/sidebar tab chrome (re-label "Tabbar" → omit; fold its swatches into the **Sidebar**
  region) or omit it. We keep **Sidebar / Tab** (the vertical-rail tab card) regions.
- **No SSH-host filter, Agents = Claude-only** — not relevant to theming/fonts; no cross-reference may
  reintroduce them in any Appearance surface.
- **NOT E15:** Recipes/snippets (E16) and Editor-section population (separate follow-up). Do not touch.
- **Golden-safe by construction.** `AppearancePreferences` and all terminal render prefs are PURE client
  chrome: never routed through `EnvBridge.toEnv`, never folded into `EnvConfig.overlay`, never written to
  `video-prefs.json`. They persist only to `UserDefaults` and drive `ThemeStore` + `TerminalConfigBroadcaster`.
  **This epic does NOT touch the wire / golden corpus.** Every `TerminalConfigBuilder` change is gated so the
  existing `nil`-override call path stays byte-identical (the regression guard for the frozen golden + the
  current builder tests).
- **Float math:** any percent/scale arithmetic (line-height multiplier→`adjust-cell-height` %, palette
  channel math) uses separate `*` then `+` — never `addingProduct`/`fma`. Clamps use ordered
  `Double.minimum`/`Double.maximum` (NaN-faithful), never bare `<`/`>` ternary. (Mirror the existing
  `CursorColorHex.channel` helper.)
- **Validate-then-drop on all parsed files.** TOML/iTerm2/Kitty/Alacritty/Ghostty parsers operate on
  untrusted user files: validate counts/lengths/hex BEFORE building, return `nil` on any malformed input,
  never force-unwrap, never trap. (Same discipline as a hostile UDP datagram.)
- **No SCStream/VT/Metal in tests.** All new logic (parsers, resolvers, builder, catalog math) is pure and
  headlessly unit-tested. UI is GUI-verified only.
- **iOS rots silently.** New cross-platform SwiftUI must compile on iOS; filesystem/`NSOpenPanel`/
  `NSWorkspace`/themes-folder code is `#if os(macOS)`. Run `bash scripts/check-ios.sh` at the end.

## Architecture decisions (host/client split constraints)

1. **Theme files live on the CLIENT, macOS only.** slopdesk renders the terminal on the HOST but
   resolves theme/chrome on the CLIENT, so custom theme files are scanned/stored client-side rather than
   on the host. Custom `.slopdesktheme` files live at `~/.config/slopdesk/themes/` on macOS. iOS has
   no `~/.config` → iOS gets built-in themes + the dual-slot toggle + all font settings, but NOT the
   folder scan / file import / Open-Folder (`#if os(macOS)` guarded; a faithful iOS document-picker
   import is explicitly deferred and noted in the UI).

2. **Custom themes need a uniform parsed model the LEAF can see.** The terminal palette must reach
   `TerminalConfigBuilder` (in `SlopDeskVideoProtocol`, the leaf), while chrome colours must reach
   `SlateTheme` (in `SlopDeskClientUI`). So the parsed theme document (`ThemeDocument`) and the
   filesystem scanner/parsers (`ThemeLibrary`) live in the **leaf** `SlopDeskVideoProtocol/Settings/` —
   pure Foundation, headlessly testable. `SlopDeskClientUI` builds a `ThemeCatalog` that combines the
   built-in `SlateTheme`s with the scanned `ThemeDocument`s and resolves a slot reference → `SlateTheme`.

3. **Dual-slot follow-OS (M2 / ES-E15-1).** Add `themeDark` + `useSeparateDarkTheme` (+ custom-slug
   per-slot) to `AppearancePreferences`. A pure resolver `ThemeResolution.activeRef(appearance:osIsDark:)`
   picks the slot; `ThemeStore` re-runs it on a macOS OS-appearance change observer so the theme switches
   **live** with the system colour scheme.

4. **Built-in themes gain a 16-colour ANSI palette.** Today `SlateTheme` carries only bg/fg + chrome. The
   swatch grid (16 dots) and "Duplicate" need a real palette. Add `ansiPalette: [String]` (16 hex) to the
   built-in Monokai Pro / Paper / Dark themes (canonical Monokai Pro ANSI values). `TerminalConfigBuilder`
   gains an OPTIONAL `paletteOverride` so the active theme's palette reaches the terminal cells; `nil` →
   no `palette` lines (byte-identical default).

5. **Font parity that has no clean libghostty key is persisted + surfaced + deferred-apply** (the exact
   precedent set by `cursorAnimation = .smooth`): underline-off and SGR blink have no stock ghostty config
   key, and only 2 of the 5 blending modes map (`default`, `macos-like`→`font-thicken`). These controls
   persist and surface with a UI note; we only emit libghostty keys verified to exist (unknown keys risk a
   config-load warning). Ligatures, fallback, bold/italic mode, and line-height DO map and are emitted.

6. **Container tokens per theme (M6).** slopdesk panes are deliberately FLAT (radius 0, Monokai Pro flat).
   We PARSE `[container]`/`[token]` and store them on `ThemeDocument`, and apply `accent` (tint) + `radius`
   for CUSTOM themes that define them. Built-in themes keep the flat design (radius 0). Shadow/border/
   padding/margin are parsed+stored, applied best-effort; full container-styling is a low-risk follow-up.

### ES-E15-1 — dual-slot follow-OS (both)
Approach: add `themeDark`/`useSeparateDarkTheme` (+ custom slugs) to `AppearancePreferences`; pure

(`adjust-cell-height`, `font-feature`, `font-style-*`/`font-synthetic-style`, `font-thicken`);

deferred ones); **GUI** — toggle ligatures/line-height, terminal reflows/re-renders (line-height triggers
SIGWINCH via the existing resize-debounce path).
