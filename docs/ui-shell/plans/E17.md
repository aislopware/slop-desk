# E17 — Read-only mode + Vi-mode pill + secure input (implementation plan)

Three per-pane terminal features:
1. **Read-only** (ES-E17-1): user-toggled per-pane input gate (keys/paste/click-to-move/mouse-report/drop), `🔒 READ ONLY ×` pill, beep-on-blocked, Shell→Read-Only menu/palette terms.
2. **Vi-mode pill** (ES-E17-2, ES-E17-3): VIEW layer over the EXISTING copy-mode engine — mode pill + live repeat-count, `⌘/` key-hint bar, `/`/`?`→find bar, line/block selection + yank (char-select is a documented libghostty ceiling).
3. **Secure input / I22** (ES-E17-4): host signals canonical no-echo → client `EnableSecureEventInput`, `SECURE INPUT` title pill, Auto/Indicator settings.

Directives: read-only IS a legitimate feature (not the agent approval-gate); secure input is macOS-only + process-global balanced; vi rides the existing copy-mode engine; the char-selection ceiling is documented not faked; no dead pills. Vertical-tabs-only — pills live in the pane top-overlay region + sidebar rail row, never a horizontal strip.

## Wire decision (justified — touches the golden corpus)

`ES-E17-4` requires Secure Keyboard Entry to engage **automatically** when the remote shell enters a hidden-password prompt. The trigger is the PTY's termios `ECHO` flag being cleared on the **host** by the child (`sudo`/`ssh`/`login`/`read -s`/`getpass`). Client-side derivation fails: `TerminalModeTracker` parses DECSET/DECRST + OSC-133 from the output byte stream, but termios `ECHO` is **not in the byte stream** — it is a host-side line-discipline attribute set via `tcsetattr`, invisible to libghostty/the client. A "Password:"-text heuristic is locale-fragile and violates the honesty rule. So AUTO **genuinely requires a new host→client message** (the carryover's sanctioned wire-extension case).

Design: new CONTROL-channel message **type 31** `inputEcho(enabled: Bool)` (host→client). 1-byte body (`enabled ? 1 : 0`), decoded as `byte != 0` (untrusted-bool rule), validate-then-drop on short body. Host emits on the ECHO edge only. Additive within wire version 1 (host accepts only v1, no negotiation) → **host + client must redeploy together** (no-backcompat). Procedure: update `docs/20-wire-protocol.md`, add a `DECISIONS.md` entry with this justification, add corevectors emission, and **hand-merge** `golden/golden_vectors.json` so the 13 frozen keys survive (`swift run slopdesk-corevectors` emits only a subset; never redirect it over the corpus). Manual big-endian binary, no JSON/Codable. The **Manual** secure-input path (Edit ▸ Secure Keyboard Entry) is client-only and ships regardless, so the feature is useful even before the wire round-trip.
